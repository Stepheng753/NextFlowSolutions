# Legal Tech Prompt Engineering: Core Specification

## 1. Role & Objective
As the **Lead AI Prompt Engineer**, the objective is to design highly robust, context-rich, and hallucination-resistant prompts for the LLMs (Google Gemini, Perplexity) operating within our n8n workflows. These prompts must translate high-level legal logic into strict, instructional frameworks that demand deterministic, highly accurate outputs for automated routing.

## 2. Standard Persona Definitions
A hallmark of our prompting strategy is establishing an authoritative, domain-specific expert persona immediately. Every prompt must define the `### Role & Objective` upfront.
- **Tone:** Professional, objective, clinical, and strictly technical. Avoid conversational filler or biased language.
- **Examples:**
  - *"You are an expert Data Normalization and Financial Reconciliation Agent."*
  - *"You are an expert Patent Analyst and Intellectual Property Strategist."*
  - *"You are an expert Legal-Medical Consultant."*

## 3. Formatting LLM Constraints
Downstream n8n nodes require predictable, structured data. To achieve this, prompts must feature a `### Constraints & Guardrails` and `### Formatting Rules` section.
- **Forcing JSON:** If the pipeline requires JSON, state explicitly: *"CRITICAL: Output ONLY valid JSON. Do not include markdown formatting (like \`\`\`json), preambles, or conversational text. The response must begin with `[` and end with `]`."*
- **Preserving Data Integrity:** Use commands like *"DO NOT DELETE DATA"* or *"DO NOT ALTER VALUES"* (e.g., map keys but maintain original string formats). If data does not fit the schema, mandate moving it to a nested `unmapped_metadata` object (a "Graveyard") rather than discarding it.
- **Structured Output Frameworks:** Use `### Few-Shot Examples` providing explicit Input/Output pairs, or mandate strict Markdown structures for textual output (e.g., forcing specific headers, bulleted lists for chronological parsing, and closed/open cross-examination rules).

## 4. Techniques for Referencing Runtime Variables (n8n Context Injection)
Instead of embedding raw n8n expression syntax directly into the prompt text, the prompts are written to define the incoming data payload semantically via a `### Core Context` section.
- **Input Definition:** Clearly explain *what* the LLM is receiving and *from where* (e.g., *"You will receive: An Invention Disclosure Form (IDF)..."* or *"The data you receive comes from an upstream OCR/Extraction agent."*). 
- **Mapping:** The actual n8n variable injection (e.g., `{{ $json.text }}`) occurs in the n8n node configuration, which simply appends the dynamic data below the prompt text fetched from the Google Doc.

## 5. Strict Anti-Hallucination Guardrails
To prevent severe legal risk and ensure deterministic behavior, every prompt enforces strict "I don't know" fallback mechanisms and grounding rules.
- **Null Mapping:** Explicitly state: *"If a specific concept is entirely missing from the raw input item, map its standard key to `null`."*
- **Grounding Rules:** Commands such as *"No Hallucinations: If citing an external medical standard, do not invent specific chapter/page numbers unless you are absolutely certain of their accuracy."* Ensure the AI relies only on the text provided for the matter.
- **No UPL (Unauthorized Practice of Law):** Prompts must explicitly instruct the LLM not to provide binding legal advice or definitive conclusions regarding legal matters, but to draft internal analyses, identify structural gaps, or prepare objective summaries for attorney review.
