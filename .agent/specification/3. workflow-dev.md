# Workflow Development Specification

## 1. Common Node Patterns
While workflows can start, end, and perform many different tasks depending on the specific use case, they frequently utilize a combination of common APIs, standard triggers, and consistent terminations. A typical, but not strictly enforced, pattern often resembles the following pipeline:
1. **Common Starts (`n8n-nodes-base.webhook`)**: Workflows typically initiate via a synchronous webhook (`POST`) expecting a response (`responseMode`: `responseNode`).
2. **Intake / Switch (`n8n-nodes-base.switch`)**: Routes traffic based on payload type (e.g., JSON text vs. binary files).
3. **Extract (`n8n-nodes-base.extractFromFile`)**: Extracts text data from binary file uploads.
4. **Format (`n8n-nodes-base.set`)**: Normalizes data types, configures or edits fields, and prepares the payload.
5. **Prompt Engine (`n8n-nodes-base.googleDocs`)**: Dynamically fetches the LLM prompt using a predetermined Google Drive file path. 
6. **LLM (`@n8n/n8n-nodes-langchain.googleGemini`)**: Ingests the formatted data and prompt to generate a secure AI response based on mocked or anonymized legal data.
7. **Common APIs (`n8n-nodes-base.httpRequest` / custom integration nodes)**: Executes third-party routing (e.g., interfacing with Clio, internal databases) when required.
8. **Common Ends (`n8n-nodes-base.respondToWebhook`)**: The workflow frequently concludes by returning a `200 OK` status with the processed JSON payload back to the frontend.

## 2. Synchronous Webhook Guidelines
Many of our frontend architectures rely on blocking, synchronous communication. When building these connections:
- **Node**: `Webhook`
- **Configuration**:
  - `httpMethod`: `POST`
  - `responseMode`: `responseNode` (Generally preferred over `onReceived` for frontend consumption)
- In these synchronous patterns, paths should eventually route to a `Respond to Webhook` node. If a synchronous pipeline finishes without sending a response, the frontend will experience a timeout.
- You should generate a strictly-typed JSON schema upon completion and save the integration details to `docs/agent-artifacts/backend-handoff.md`, especially when integrating with custom frontends.
- While asynchronous polling loops can be engineered, synchronous responses are typically requested by our blueprints unless otherwise noted.

## 3. Recommended Error-Trigger Routing Rules
Because webhooks are synchronous, errors should be caught gracefully and relayed to the frontend and internal alerting mechanisms to avoid leaving the client hanging.
When crafting complex workflows, consider including an isolated error-handling branch:
1. **Error Trigger (`n8n-nodes-base.errorTrigger`)**: Automatically executes if any node in the main pipeline fails.
2. **Alerting (`n8n-nodes-base.discord` / `slack`)**: Sends a detailed error alert indicating the exact workflow and node failure to the backend engineering team.
3. **Error Response (`n8n-nodes-base.respondToWebhook`)**:
   - `respondWith`: `json`
   - `responseCode`: `500` (or `400` depending on context)
   - `responseBody`: 
     ```json
     {
       "error": "Pipeline failed.",
       "details": "{{ $json.error.message }}"
     }
     ```
This ensures the frontend API consumer receives a structured error body instead of indefinitely waiting.

## 4. Updates & Maintenance
This document is the absolute source of truth for the n8n backend architecture. It must be updated immediately after any major change to our node pipeline structure, data routing methods, or error-handling standards.
