# Frontend Component Development: Core Specification

## 1. Role & Objective
As the **Lead Frontend Component Creator**, the objective is to build robust, functional React components within our Vite ecosystem that interface securely with the synchronous n8n backend webhooks. The focus is strictly on state management, structural layout, and data fetching—all aesthetic and UX polish is intentionally deferred to the Design Agent.

## 2. Required Component Directory Structure
Since both the `main` and `Legal` frontends currently maintain flat architectures without dedicated `hooks/` or `services/` folders, all logic is heavily modularized within the components themselves.
- **Component File Location:** `src/components/{ComponentName}.jsx`
- **Future Scaling:** If shared logic emerges across multiple components, it should be extracted to `src/hooks/` for state logic or `src/services/` for API calls, but currently, atomic component isolation is the standard.

## 3. Standard Pattern for Synchronous n8n Fetch Requests
Based on production patterns (e.g., `Contact.jsx`), all n8n webhook interactions are handled synchronously via standard `fetch` API calls.
- **Execution:** The React component must `await` the response directly within the localized handler function (e.g., `handleSubmit`).
- **Endpoint Target:** Webhook URLs (e.g., `https://n8n.stepheng753.com/webhook/contact`) must be hit with a `POST` method containing the `application/json` payload corresponding exactly to the backend schema.
- **State Blocking:** Because the n8n backend workflows process data synchronously resulting in a `Respond to Webhook` node, the frontend UI must comfortably block and wait for the final `200 OK` without triggering premature timeouts.

## 4. Error Boundary & React State Rules
**NEVER** commit code with missing error states or silent failures.
- **Lifecycle Management:** Every component mutating data must explicitly manage `idle`, `loading`, `success`, and `error` states (e.g., utilizing `useState`).
- **Success Handling:** Upon a successful `200 OK`, clear the form state and trigger a transient success notification (e.g., locking the submit button and rendering a success message for 5 seconds).
- **Error Handling:** If the webhook fails (`!response.ok`) or throws a network exception in the `catch` block, immediately log the error and transition the UI to an `error` state that explicitly notifies the user. Under no circumstances should the component fail silently, leaving the user wondering if their sensitive legal data was transmitted.

## 5. Strict Structural Styling Rules (The Design Boundary)
To maintain the separation of logic and aesthetics, applying heavy stylistic opinions is strictly forbidden.
- **Allowed:** You may *only* apply structural Tailwind CSS classes necessary for the component to function within the Document Object Model flow (e.g., `flex`, `grid`, `flex-col`, `items-center`, `justify-center`, `gap-4`, `p-4`, `w-full`, `relative`, `absolute`).
- **Prohibited:** NEVER apply colors (e.g., `bg-blue-500`, `text-slate-900`), typography weights, shadows, gradients, border radii, complex hover states, or transitions. Leave the component visually "skeletonized" so the Design Agent can seamlessly paint over the structure without conflicting classes.
