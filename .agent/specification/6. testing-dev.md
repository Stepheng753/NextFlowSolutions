# 6. testing-dev: QA Automation & UI Testing Specification

As the QA Automation Engineer, this document defines the strict standards for writing Playwright E2E and React component tests within the Nextflow application. It specifically outlines the required structure for mocking synchronous n8n webhooks and handling legal data safely.

## 1. Directory Structure & Mock Data Storage

All tests and testing-related resources must be organized under the `/tests` directory at the project root.

- `/tests/e2e/`: Contains end-to-end tests for full application flows.
- `/tests/components/`: Contains Vitest/Playwright component tests.
- `/tests/fixtures/`: The mandatory location for all mock data.

### Rule: Storing Mock Legal Data

**NEVER** use real client data. All mock legal data MUST be synthetic and securely stored in JSON files within `/tests/fixtures/`. 
For example, create `/tests/fixtures/mock-legal-data.json` for standardized dummy payloads. Tests should actively import or read this JSON to inject into their mocks.

## 2. Absolute Prohibition of Hitting Live APIs

Under **no circumstances** should an automated test hit a live n8n webhook or production external API.

1. **Strict Mocking:** You must intercept all network requests dispatched to n8n webhooks (typically matching `**/webhook/**` or `**/webhook-test/**`).
2. **Data Injection:** Inject the mock legal data from the `/tests/fixtures/` directory to simulate both successful and failed webhook responses.

```typescript
// Example of strict network interception
await page.route('**/webhook/**', async (route) => {
  // Read and return mock data payload
  const mockPayload = require('../fixtures/mock-legal-data.json');
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify(mockPayload.successResponse),
  });
});
```

## 3. Standard Playwright Test Block Structure

Every Playwright E2E test must follow a predictable, highly-structured `test.describe` and `test` block pattern, ensuring route interception is set up before UI interaction.

```typescript
import { test, expect } from '@playwright/test';

test.describe('Legal Document Intake Flow', () => {
    test.beforeEach(async ({ page }) => {
        // 1. Setup global mocks or navigate to the base page
        await page.goto('/intake');
    });

    test('should successfully submit formal legal request and display success banner', async ({ page }) => {
        // 2. Intercept and mock the n8n webhook
        await page.route('**/webhook/submit-intake', async (route) => {
            await route.fulfill({ status: 200, body: JSON.stringify({ success: true, referenceId: 'LEG-1234' }) });
        });

        // 3. Perform UI interactions
        await page.fill('input[name="clientName"]', 'John Doe');
        await page.click('button[type="submit"]');

        // 4. Assert UI state (Banners)
        await expect(page.locator('.success-banner')).toBeVisible();
        await expect(page.locator('.success-banner')).toContainText('LEG-1234');
    });
});
```

## 4. Required Assertions for n8n Webhook Responses

Because the frontend relies heavily on synchronous n8n webhooks, verifying the UI's reaction strictly involves testing the presence, visibility, and content of success and error banners.

- **Success Banners:** When simulating a `200 OK` from a webhook, you MUST assert that the success banner (e.g., `.success-banner` or specific Tailwind alert classes) becomes visible and contains the expected return payload text (such as a reference ID or summary).
- **Error Banners:** When simulating a `4xx` or `5xx` error from a webhook, you MUST assert that the error banner (e.g., `.error-banner`) is shown to the user, ensuring the unhandled exception or rejected status is proactively mapped to a localized error message in the UI.

```typescript
test('should display error banner on synchronous n8n validation failure', async ({ page }) => {
    // 1. Mock a failed n8n webhook response
    await page.route('**/webhook/submit-intake', async (route) => {
        await route.fulfill({ 
            status: 400, 
            body: JSON.stringify({ error: 'Missing mandatory confidentiality clause.' }) 
        });
    });

    // 2. Trigger the submission
    await page.click('button[type="submit"]');

    // 3. Strict Assertion for Error State
    await expect(page.locator('.error-banner')).toBeVisible();
    await expect(page.locator('.error-banner')).toContainText('Missing mandatory confidentiality clause');
});
```

## Summary Checklist
- [ ] Are all mock payloads stored in `/tests/fixtures/`?
- [ ] Is `page.route` used to completely isolate the test from live n8n workflows?
- [ ] Are there specific UI assertions validating both success and error banners?
